# FR-145 ŌĆö HyperLogLog++ (HLL++)

## Overview
The original HyperLogLog (FR-144) has a documented bias for small cardinalities (the empirical "small-range correction" overshoots for `n < 5m`). HLL++ replaces the heuristic correction with a learned bias table, switches to 64-bit hashes (extending range to 2^64 instead of 2^32), and adds a sparse representation that uses far less memory at low cardinalities. FR-145 supersedes FR-144 as the production-grade unique-visitor counter while keeping the same per-page register footprint at saturation.

## Academic source
Heule, S., Nunkesser, M., and Hall, A. "HyperLogLog in practice: algorithmic engineering of a state of the art cardinality estimation algorithm." *Proceedings of the 16th International Conference on Extending Database Technology (EDBT '13)*, pp. 683ŌĆō692, 2013. DOI: 10.1145/2452376.2452456.

## Formula
Identical register update to HLL: `M[j] ŌåÉ max(M[j], Žü(w))`. The estimator changes:

**1. Raw estimate** (same as HLL):
```
E = ╬▒_m ┬Ę m┬▓ ┬Ę ( ╬Ż 2^{ŌłÆM[j]} )^{ŌłÆ1}
```

**2. Empirical bias correction:** for `E Ōēż 5m`, look up a precomputed bias table `bias(E, p)` (provided in the paper for `p Ōłł [4, 18]`) and subtract:

```
E' = E ŌłÆ bias(E, p)
```

The table is generated by Monte Carlo: for each grid `E_target`, simulate many HLL runs achieving that estimate, measure mean error, store as bias offset.

**3. Range selection rule:**
```
n╠é = E_linear_counting          if E_raw Ōēż 2.5 ┬Ę m and V > 0
n╠é = E'                         if E_raw Ōēż threshold(p)
n╠é = E                          otherwise
```

where `E_linear_counting = m ┬Ę ln(m / V)` (linear counting estimator), `V` = number of zero registers, and `threshold(p)` is a per-precision empirical cutoff (table 4 in the paper, e.g., threshold(12) = 6,500; threshold(14) = 22,500).

**4. Sparse representation:** below cardinality `m/4`, store register updates as a list of `(j, Žü(w))` pairs in a sorted array of variable-length-encoded 32-bit ints. Convert to dense (m bytes) when sparse encoding exceeds dense size.

## C++ implementation
- File: `backend/extensions/hyperloglog_pp.cpp`
- Entry: `void hllpp_add(HllPpState* state, uint64_t hash)`, `double hllpp_estimate(const HllPpState* state)`
- Complexity: O(1) amortised per insertion (sparse-to-dense conversion is amortised). O(m) for dense estimate; O(s log s) for sparse estimate (s = sparse list length).
- Thread-safety: per-page state. SIMD: `╬Ż 2^{ŌłÆM[j]}` SIMD'd; bias-table lookup is scalar. Memory: dense path m bytes; sparse path up to m/4 bytes (Ōēż 4 KB at p=14).

## Edge cases & neutral fallback
Empty state ŌåÆ estimate = 0. Sparse-to-dense conversion at threshold; both directions supported (paper requires sparse ŌåÆ dense only). Bias table missing for arbitrary `p` ŌåÆ fall back to original HLL correction. NaN/empty input items ŌåÆ skip. Saturation: 64-bit hash with `p Ōēż 18` and `Žü(w) Ōēż 64 ŌłÆ p`; saturation point is ~2^54, effectively infinite for forum data.

## Minimum-data threshold
None ŌĆö HLL++ is accurate from cardinality 0 upward (linear counting handles the small-range case).

## Budget
Disk: Ōēż16 KB/page (dense at p=14) or Ōēż4 KB (sparse) ┬Ę  RAM: <400 MB total for 25,000 pages at p=14 dense

## Scope boundary vs existing signals
FR-145 supersedes FR-144 (HLL) ŌĆö same purpose, better accuracy. Both exist as separate FRs because some operators may want the simpler classic HLL for compatibility with external systems (Redis, Postgres `cardinality_estimators`). FR-148 (Space-Saving) is unrelated: it tracks top-k items, not cardinality.
